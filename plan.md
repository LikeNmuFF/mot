# Xeia — 18th Monthsary App (for Erica Joy)

A personal Android mobile app that tells our story — part surprise, part keepsake. Built as a React web app wrapped with **Capacitor** into a real installable `.apk`, built via **GitHub Actions CI/CD** (no local Android Studio required). Fully responsive UI, works offline after first install.

**Timeline:** 5 days
**Stack:** React (Vite) + Tailwind CSS + Framer Motion + Capacitor (Android) + GitHub Actions
**Build target:** Android `.apk`, built in CI, downloaded as a workflow artifact and sideloaded onto her phone

---

## 1. Concept

First open feels like an event — animated intro, scroll-through timeline of our relationship, one interactive moment, then a finale message. Every open after that lands on a "keepsake" home screen where she can revisit the timeline, replay the finale, and see a live running counter of time together.

State (first-open vs. return-visit) is tracked via `localStorage` — no backend, no login, no server needed.

## 3. Tech Stack & Setup

- **Vite** — fast dev/build, lighter than CRA
- **React 18**
- **Tailwind CSS** — utility-first styling for speed
- **Framer Motion** — animations (fades, slides, staggered entrances)
- **vite-plugin-pwa** — generates manifest + service worker for installability and offline support
- **No backend** — all content is local (JSON/JS data files + bundled images)
- **Camera access** — native browser `getUserMedia` API for the Photobooth, accessed through Capacitor's WebView (works the same as browser camera access; Capacitor's Camera plugin is an optional upgrade for more native-feeling permission prompts)
- **Capacitor** — wraps the built React app into a native Android project, producing a real `.apk`
- **GitHub Actions** — CI workflow builds the Android app (Java + Android SDK + Gradle, all installed in CI, nothing local) and outputs a downloadable `.apk` artifact
- **No Vercel/Netlify needed** — since this is now a native app, not a hosted website; all content ships inside the APK itself

### Initial setup commands
```bash
npm create vite@latest xeia -- --template react
cd xeia
npm install
npm install -D tailwindcss postcss autoprefixer
npx tailwindcss init -p
npm install framer-motion
npm install @capacitor/core @capacitor/android
npm install -D @capacitor/cli
npx cap init xeia com.erica.xeia --web-dir=dist
npm run build
npx cap add android
npx cap sync android
```

## 6. Screen-by-Screen Features

### LoginScreen (new — first screen, every visit)
- A simple "lock" screen shown before anything else, every time the app opens (not just first visit)
- Prompt: something like "When did it all start?" with a date input (or 3 separate fields: month/day/year, or a styled dropdown — friendlier on mobile than a raw date picker)
- She enters January 24, 2025 — compared client-side against `content.unlockDate`
- Correct answer → soft unlock animation → proceeds to IntroScreen (first visit) or HomeScreen (return visit)
- **Wrong answer → app stays locked.** No hint, no bypass — just a gentle shake animation and the fields reset so she can try again. She simply cannot proceed until the exact date is entered correctly.
- No real authentication, no accounts, no backend — this is still just a playful client-side gate for intimacy, not real security (anyone could technically view the source and find the date). Skip this screen automatically in local dev testing via a query param or dev flag so you're not re-entering the date on every refresh while building.

### IntroScreen
- App title / her name, soft fade-in animation
- Live counter (months, days, hours since start date)
- "Tap to begin" CTA
- Only shown on first open (checked via localStorage flag)

### Timeline
- Vertical scroll, one TimelineCard per milestone
- Staggered fade/slide-in as each card enters viewport (Framer Motion `whileInView`)
- Photo, date, title, caption per card
- "Continue" button at the end leading to InteractiveMoment

### InteractiveMoment (pick ONE for the 5-day scope)
- **Option A — Spin wheel:** "What we're doing to celebrate" — lowest build risk
- **Option B — Slider:** playful "how much do you love me" maxing out — simplest, high charm
- **Option C — Memory match game:** uses real photos, most personal but highest build time
- Recommended for 5-day timeline: **A or B**

### Gallery
- Grid view of all your photos (from `gallery.js`), separate from the curated timeline
- Tap a photo to view full-screen with a soft tulip-accented frame
- Simple swipe/tap to move between photos in full-screen view
- Accessible both from the main flow and from HomeScreen on return visits

### Photobooth
- Uses the device camera (`navigator.mediaDevices.getUserMedia`) — works in mobile browsers, no native app needed
- She takes a sequence of photos (classic photobooth style — e.g. 3 or 4 shots with a countdown between each)
- Captured frames are composited via `<canvas>` into a single templated **photo strip** — tulip-themed border/frame, small "Xeia" or date watermark, similar to a real photobooth strip
- Result can be saved to her device (canvas → downloadable image) and optionally added into the in-app Gallery so it becomes part of your collection
- Needs a graceful fallback/message if camera permission is denied or unavailable (some in-app browsers restrict camera access — test this specifically, e.g. if she opens the link inside Messenger/Instagram's in-app browser rather than Safari/Chrome directly)

### FinaleScreen
- Big reveal: message text (or embedded short video/audio)
- Optional confetti/particle burst animation on entrance
- "Save to your home screen" prompt appears here if not yet installed
- Button to go to HomeScreen

### HomeScreen (return visits only)
- Title + live counter (always updating)
- Buttons: "Replay our story" (timeline), "Read your letter again" (finale), "Our gallery" (Gallery), "Photobooth" (Photobooth), optionally "Play again" (interactive moment)
- This is the default landing screen after the very first visit

## 7. Navigation / State Logic

- No router library needed — a single `useState` in `App.jsx` tracks current screen: `"login" | "intro" | "timeline" | "interactive" | "finale" | "home" | "gallery" | "photobooth"`
- App always starts at `"login"` — she must enter the date every time she opens it (this is intentional; it's part of the charm, not just a first-visit gate)
- After correct unlock, check `localStorage.getItem("hasOpenedBefore")`:
  - Not set → go to `"intro"`, and set the flag once she reaches `"finale"`
  - Set → go to `"home"` directly
- `"gallery"` and `"photobooth"` are reachable from HomeScreen at any time, with a back button returning to `"home"`
- Live counter logic (used in IntroScreen + HomeScreen) calculates elapsed time from `content.startDate` to `Date.now()`, formatted as months/days/hours

## 8. Visual Direction

- **Theme:** tulips — since she loves them, weave this in as the recurring visual motif rather than generic hearts: a tulip icon for loading states, a soft tulip line-drawing watermark on the photobooth strip, tulip-colored accents (soft pink, or purple/lilac tulip tones) instead of default red/pink romantic clichés
- **Palette:** warm, soft — dusty rose / cream / muted gold, or a lilac + cream palette that plays off tulip colors. Avoid generic red/pink Valentine's clichés.
- **Typography:** one serif for headings/names (love-letter feel), one clean sans-serif for body text
- **Motion:** gentle fades/slides only — intimate, not flashy or "demo-like"
- **Photos:** full-bleed or soft rounded frames with subtle shadow
- **Layout:** mobile-first single column, generous whitespace, large tap targets
- **Detail touches:** subtle heartbeat pulse on the counter digits, soft particle burst on finale reveal, a tulip motif on the photobooth strip border

### Responsive design
- Mobile-first build (this is how she'll open it), but layouts must scale cleanly up to tablet and desktop widths too, in case you show it to her on a laptop or she reopens it later on a bigger screen
- Use Tailwind's responsive breakpoints (`sm:`, `md:`, `lg:`) throughout rather than fixed pixel widths
- Gallery grid: 2 columns on mobile, 3-4 columns on tablet/desktop
- Photobooth camera view: full-width on mobile, constrained/centered max-width on larger screens so the camera feed doesn't stretch oddly
- Test on at least: a real phone browser, a tablet-sized viewport (or browser resize), and desktop

## 8b. Design & Motion Guidelines (for opencode to follow while building)

The goal is for Xeia to feel like it was designed specifically for you two — not a generic template with hearts and a countdown slapped on. A few principles to hold throughout the build:

**One signature moment, not scattered effects**
Pick a single "wow" moment to spend real polish on — most likely the finale reveal, or the first unlock after LoginScreen. Everything else (timeline cards, buttons, transitions) should be quiet, consistent, and disciplined by comparison. Don't animate everything equally; that reads as templated, not intentional.

**Typography with personality**
Don't default to system fonts. Pair one characterful serif (for her name, the intro line, the finale message — anywhere it should feel handwritten/personal) with one clean sans-serif (for captions, UI labels, buttons). Import both via Google Fonts. This pairing does more to make Xeia feel bespoke than almost anything else.

**Color discipline**
Stick to the palette actually derived from the tulip logo — the purple/lilac gradient tones, cream background, muted green (from the leaf) as a tertiary accent. Don't add generic pink/red "romantic app" colors on top of it; let the logo's palette be the single source of truth for the whole UI, not just the icon.

**Motion should mean something**
- Timeline cards: gentle fade + slight upward slide as she scrolls to them (Framer Motion `whileInView`) — this rewards scrolling, don't overdo the distance/speed
- LoginScreen wrong entry: a real shake (quick, small amplitude) so it reads as "try again," not broken
- Finale reveal: this is the one place to be a little more generous with animation — a soft entrance, maybe the tulip logo motif appearing before the message
- Everything else (buttons, nav): subtle, fast (150-250ms), no bounce/spring overkill
- Respect `prefers-reduced-motion` — fall back to simple fades if she has that setting on

**Structure should encode real content**
If the timeline uses numbered markers or dates as structural devices, that's justified because it's a real chronological sequence — good. Don't add decorative numbering or icons elsewhere just for visual interest; every structural element should represent something true about your actual relationship content.

**Restraint check before shipping**
Before Day 5 polish, look at the whole flow once more and ask: is there one element doing too much (too many effects stacked on one screen), or one screen that feels flat compared to the rest? Cut or add accordingly. A calm, cohesive app beats a flashy inconsistent one.

## 9. Native Build via GitHub Actions (No Local Android Studio)

Since you don't have Android Studio locally, the entire Android build happens in CI. Here's the workflow:

**`.github/workflows/build-android.yml`** (conceptual outline — opencode should generate the full working version):
```yaml
name: Build Android APK
on:
  push:
    branches: [main]
  workflow_dispatch:        # lets you trigger a build manually from GitHub's UI

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm install
      - run: npm run build              # builds the React app into /dist
      - run: npx cap sync android        # syncs web build into the native Android project
      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'
      - name: Build APK
        run: |
          cd android
          ./gradlew assembleDebug
      - uses: actions/upload-artifact@v4
        with:
          name: xeia-apk
          path: android/app/build/outputs/apk/debug/app-debug.apk
```

**How you get the APK:**
1. Push your code to GitHub (or trigger manually via `workflow_dispatch` in the Actions tab)
2. Wait for the workflow to finish (usually a few minutes)
3. Download the `xeia-apk` artifact from the completed workflow run
4. Transfer the `.apk` to her phone (Google Drive, direct USB transfer, or a messaging app)
5. She'll need to allow **"Install from unknown sources"** for the app that's opening the file (a one-time Android setting) since this isn't from the Play Store

**Signing note:** `assembleDebug` produces an unsigned debug build, which is fine for personal sideloading — no Play Store submission needed, no signing keys to manage. This keeps the CI setup simple and avoids a whole category of keystore/signing complexity you don't need for a personal gift.

**Known risks with this approach (be aware going in):**
- Gradle/Android SDK version mismatches in CI are the most common failure point — if the first run fails, check the Actions log for the exact version conflict and pin versions accordingly
- Each CI run takes a few minutes, so iterating on native-specific issues (camera permissions, icons, splash screen) is slower than local development — do as much testing as possible on the *web* version first (via `npm run dev` in a mobile browser) before worrying about native-specific bugs
- Camera permissions on native Android require a permission entry in `android/app/src/main/AndroidManifest.xml` (`<uses-permission android:name="android.permission.CAMERA" />`) — this is easy to miss and will silently fail the Photobooth if skipped

## 10. Phased Build Plan (5 Days)

**Scope note:** Going native via Capacitor + GitHub Actions adds real setup risk on top of Login, Gallery, and Photobooth. With 5 days total, protect your deadline by building and testing everything as a **web app first** (in a mobile browser via `npm run dev`), and only wrapping it with Capacitor once the app itself is done — native build issues are much easier to debug against a working app than a half-built one.

### Phase 1 — Day 1: Foundation + Content
- [ ] Scaffold Vite + React + Tailwind + Framer Motion
- [ ] Gather all real content: photos, captions, exact dates, finale message
- [ ] Fill in `timeline.js`, `gallery.js`, and `content.js` with real data
- [ ] Set up Tailwind responsive breakpoints + tulip color palette

### Phase 2 — Day 2: Login + Intro + Timeline
- [ ] Build `LoginScreen.jsx` (date input, validation against `content.unlockDate`, shake + reset on wrong entry, unlock animation on correct entry)
- [ ] Build `Counter.jsx` (live elapsed-time calculation, reusable)
- [ ] Build `IntroScreen.jsx` with entrance animation + counter + CTA
- [ ] Build `TimelineCard.jsx` + `Timeline.jsx` with scroll-triggered animations, responsive across breakpoints
- [ ] Wire state flow: login → intro → timeline
- [ ] Test everything so far in a mobile browser (not yet wrapped in Capacitor)

### Phase 3 — Day 3: Interactive Moment + Finale + Home
- [ ] Build chosen `InteractiveMoment.jsx` (spin wheel or slider)
- [ ] Build `FinaleScreen.jsx` (message/video/audio + confetti)
- [ ] Build `HomeScreen.jsx` (return-visit keepsake view, links to all sections)
- [ ] Implement localStorage first-open/return-visit routing
- [ ] Polish all screen transitions so far

### Phase 4 — Day 4: Gallery + Photobooth + Capacitor Wrap
- [ ] Build `Gallery.jsx` — responsive grid + full-screen photo view
- [ ] Build `Photobooth.jsx` — camera access, capture sequence with countdown
- [ ] Build `PhotoStripTemplate.jsx` — canvas compositing into a tulip-themed strip, downloadable image
- [ ] Wire Gallery + Photobooth into HomeScreen navigation
- [ ] Run `npx cap init` + `npx cap add android`, generate icon/splash assets
- [ ] Add camera permission to `AndroidManifest.xml`
- [ ] Set up `.github/workflows/build-android.yml`

### Phase 5 — Day 5: CI Build, Install, Test, Fix
- [ ] Push to GitHub, trigger the Actions build, download the APK artifact
- [ ] Install APK on her actual phone (or a test Android device) via "unknown sources"
- [ ] Test full flow natively: login, timeline, camera/photobooth permissions specifically
- [ ] Fix any CI/native-specific issues (this is the highest-risk step — leave real buffer time here)
- [ ] Final run-through as if seeing it for the first time, on the actual device
